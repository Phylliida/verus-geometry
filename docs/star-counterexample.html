Why the Star Polygon Breaks the Lemma

A counterexample to: local convexity ⟹ vertex ∈ convex polygon

01 The Setup

The Verus codebase defines is_convex_polygon using a local condition: every three consecutive vertices must not make a right turn. Formally, for every index i, the orientation of the triple (v[i], v[i+1], v[i+2]) must be non-negative.

The lemma lemma_vertex_in_convex_polygon claims that if a polygon is "convex" by this definition, then every vertex satisfies point_in_convex_polygon_boundary_inclusive — meaning you can't find one edge where the vertex is on the left and another where it's on the right.

This sounds obviously true for a normal convex polygon. But the local definition allows something sneaky: a star polygon that self-intersects while always turning left.

02 The Star Polygon

Take five points arranged like a regular pentagon, but visit them in a star order — skipping every other vertex. The polygon visits: A → D → B → E → C → back to A.

Star edges Turn arcs

In the star traversal, the path crosses itself — edges AD and BE intersect near the center. Yet at every vertex, the turn is to the left. The polygon winds around twice (winding number 2), and each step is a ~144° left turn.

03 Local Convexity Check: All Passes

The orient2d function computes a 2×2 determinant: positive means a left turn, negative means right, zero means collinear. Let's check every consecutive triple:

orient2d(A, D, B) = +197 ← left turn ✓
orient2d(D, B, E) = +198 ← left turn ✓
orient2d(B, E, C) = +198 ← left turn ✓
orient2d(E, C, A) = +197 ← left turn ✓
orient2d(C, A, D) = +180 ← left turn ✓
✓ is_convex_polygon passes — every consecutive triple is non-negative.

04 The Failure: Vertex A vs. All Edges

Now test vertex A = (0, 10) against every edge. The point_in_convex_polygon_boundary_inclusive predicate requires that we don't see both positive and negative signs:

Positive orient2d (A is left of edge) Negative orient2d (A is right of edge) Zero orient2d (A on edge)
orient2d(A, D, A) = 0 ← A is endpoint of this edge
orient2d(D, B, A) = +197 ← A is to the LEFT
orient2d(B, E, A) = -126 ← A is to the RIGHT ✗
orient2d(E, C, A) = +126 ← A is to the LEFT
orient2d(C, A, A) = 0 ← A is endpoint of this edge
✗ Both positive AND negative signs found → point_in_convex_polygon_boundary_inclusive(A, polygon) is FALSE.

Edge D→B has A on its left (+197), but edge B→E has A on its right (−126). The predicate sees mixed signs and reports that A is not inside the polygon — even though A is a vertex of the polygon. The lemma's postcondition fails.

05 Why It Happens

Half-plane where orient2d > 0 Half-plane where orient2d < 0

Every directed edge divides the plane into two half-planes: a "left" side (positive orient2d) and a "right" side (negative). For a truly convex polygon, all vertices sit in the left half-plane of every edge. But in the star polygon, vertex A is on the left of edge D→B and on the right of edge B→E. The self-intersection lets a vertex escape to the "wrong" side of a far-away edge.

The core issue: Local convexity (consecutive triples only) does not prevent self-intersection. A star polygon can turn left at every vertex while its edges cross each other. This breaks the assumption that vertices stay on a consistent side of all edges — which is what the point_in_convex_polygon_boundary_inclusive predicate actually checks.

06 The Fix

To make lemma_vertex_in_convex_polygon true, the definition needs a global condition — either directly (every vertex is left of every edge) or by adding a simplicity requirement (no self-intersections). With the current local-only definition, star polygons slip through and the lemma is provably false.
